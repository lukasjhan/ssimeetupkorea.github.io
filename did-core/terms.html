이 섹션은 이 규격과 탈중앙 식별자 인프라 전체에서 사용되는 용어를 정의한다. 이 규격에서 이 용어들이 나타날 때마다 이 용어들에 대한 링크를 포함한다.

증폭 공격(amplification attack)
공격자가 시스템에 작고 유효한 입력을 제공하여 대상 시스템의 CPU, 저장소, 네트워크 또는 기타 자원을 고갈시키려는 공격 유형이다. 이러한 입력은 그 자체보다 처리하는 데 비용이 기하급수적으로 많이 드는 해로운 효과를 초래한다.
인증(authenticated)
인증은 개체가 하나 이상의 검증 방법을 사용하여 특정 속성을 가지고 있거나 특정 비밀을 제어한다는 것을 증명하는 과정이다. DID에서 일반적인 예는 DID 문서에 게시된 공개키와 연관된 암호화 개인키의 제어를 증명하는 것이다.
바인딩(binding)
DID 변환기 또는 DID URL 역참조자를 호출하기 위해 호출자가 사용하는 구체적인 메커니즘이다. 이는 로컬 명령줄 도구, 소프트웨어 라이브러리 또는 HTTPS 요청과 같은 네트워크 호출일 수 있다.
암호화 스위트(cryptographic suite)
특정 암호화 기본 요소의 사용을 정의하는 규격으로 특정 보안 목표를 달성하기 위해 사용된다. 이 문서들은 종종 검증 방법, 디지털 서명 유형, 식별자 및 기타 관련 속성을 지정하는 데 사용된다.
탈중앙 식별자 (DID)
중앙 집중식 등록 기관이 필요 없는 전 세계적으로 고유한 영구 식별자이다. 이는 종종 암호화적으로 생성 및/또는 등록된다. DID의 일반적인 형식은 에서 정의된다. 특정 DID 체계는 DID 메소드 규격에서 정의된다. 많은 DID 방법들이 분산 원장(DLT) 또는 기타 형태의 탈중앙 네트워크를 사용하지만 모든 DID 방법들이 그런 것은 아니다.
탈중앙 신원 관리(decentralized identity management)
신원 관리는 탈중앙 식별자의 사용을 기반으로 한다. 탈중앙 식별자 신원 관리는 식별자 생성, 등록 및 할당에 대한 권한을 X.500 디렉토리 서비스, 도메인 이름 시스템, 대부분의 국가 ID 시스템과 같은 전통적인 신뢰점을 넘어서 확장한다.
탈중앙 공개 키 인프라 (DPKI)
공개 키 인프라는 전통적인 인증 기관에 의존하지 않으며, 탈중앙 식별자와 DID 문서를 사용하여 공개 키 설명을 발견하고 검증한다.
DID 컨트롤러
DID 문서에 변경을 가할 수 있는 엔티티이다. DID는 여러 DID 컨트롤러를 가질 수 있다. DID 컨트롤러는 DID 문서의 최상위에 선택적 `controller` 속성을 통해 명시될 수 있다. DID 컨트롤러는 DID 주체일 수도 있다.
DID 대리인
DID 컨트롤러가 DID에 관련된 검증 방법을 사용할 수 있는 권한을 부여한 엔티티이다. 예를 들어, 자녀의 DID 문서를 관리하는 부모는 자녀가 개인적인 장치를 사용하여 인증할 수 있도록 허용할 수 있다. 이 경우, 자녀는 DID 대리인이다. 자녀의 개인 장치에는 자녀가 DID를 사용하여 인증할 수 있도록 하는 개인 암호화 자료가 포함될 것입니다. 그러나 부모의 허락 없이 다른 개인 장치를 추가할 수는 없을 것이다.
DID 문서
DID 주체를 설명하는 데이터 세트로, DID 주체 또는 DID 대리인이 자신을 인증하고 DID와의 연관성을 증명할 수 있는 메커니즘, 예를 들어 암호화된 공개 키와 같은 것들을 포함한다. DID 문서는 또는 W3C DID 사양 레지스트리 [[DID-SPEC-REGISTRIES]]에 정의된 대로 하나 이상의 다른 표현을 가질 수 있다.
DID 프레그먼트
DID URL에서 첫 번째 해시 기호(#) 뒤에 오는 부분입니다. DID 프레그먼트 구문은 URI 프레그먼트 구문과 동일합니다.
DID 메소드
특정 DID 체계가 구현되는 방법에 대한 정의이다. DID 메소드는 DID 메소드 규격에 의해 정의되며, DID와 DID 문서가 생성, 해석, 갱신, 그리고 비활성화되는 정확한 작업들을 명시한다. 참조.
DID 경로
DID URL에서 첫 번째 슬래시(/) 기호로 시작하고 물음표(?) 기호, 프래그먼트 해시 기호(#) 또는 DID URL의 끝으로 끝나는 부분이다. DID 경로 구문은 URI 경로 구문과 동일하다. 참조.
DID 쿼리
DID URL에서 첫 번째 물음표 기호(?)를 포함하여 그 뒤를 따르는 부분이다. DID 쿼리 구문은 URI 쿼리 구이과 동일하다. 참조.
DID 해석
DID와 해석 옵션들의 집합을 입력으로 받아, 규범을 준수하는 표현으로 DID 문서와 추가 메타데이터를 반환하는 과정이다. 이 과정은 해당 DID 메소드의 "읽기" 작업에 의존한다. 이 과정의 입력과 출력은 에서 정의된다.
DID 변환기(DID resolver)
DID 변환기는 DID를 입력으로 받아 규범을 준수하는 DID 문서를 출력으로 생성하는 DID 해석 기능을 수행하는 소프트웨어 및/또는 하드웨어 구성요소이다.
DID 체계(DID scheme)
탈중앙 식별자의 공식 구문이다. 일반적인 DID 체계는 에 정의된 대로 did: 접두사로 시작합니다. 각 DID 메소드 규격은 해당 DID 메소드와 함께 작동하는 특정 DID 메소드 체계를 정의한다. 특정 DID 메소드 체계에서는 DID 메소드 이름이 첫 번째 콜론 다음에 오고 두 번째 콜론으로 끝나며, 예를 들면 did:example:와 같다.
DID 주체(DID subject)
DID에 의해 식별되고 DID 문서에 의해 설명되는 엔티티이다. DID 주체는 어떤 것이든 될 수 있다: 사람, 그룹, 조직, 물리적 사물, 디지털 사물, 논리적 사물 등.
DID URL
에 정의된 대로 추가적인 구문적 구성요소를 포함하는 DID입니다. 이것은 선택적인 DID 경로 (앞에 오는 / 문자 포함), 선택적인 DID 쿼리 (앞에 오는 ? 문자 포함), 그리고 선택적인 DID 프레그먼트 (앞에 오는 # 문자 포함)를 포함한다.
DID URL 역참조(DID URL dereferencing)
DID URL과 입력 메타데이터 집합을 입력으로 받아 리소스를 반환하는 과정이다. 이 리소스는 추가 메타데이터가 포함된 DID 문서, DID 문서 내에 포함된 보조 리소스, 또는 DID 문서 외부의 완전히 다른 리소스일 수 있다. 이 과정은 DID URL 내에 포함된 DID에 의해 표시된 DID 문서를 가져오기 위해 DID 해석을 사용한다. 역참조 과정은 DID 문서에 추가 처리를 수행하여 DID URL에 의해 표시된 역참조된 리소스를 반환할 수 있다. 이 과정의 입력과 출력은 에서 정의된다.
DID URL 역참조자(DID URL dereferencer)
주어진 DID URL 또는 DID 문서에 대한 DID URL 역참조 기능을 수행하는 소프트웨어 및/또는 하드웨어 시스템이다.
분산 원장(distributed ledger) (DLT)
사건을 기록하기 위한 비중앙화된 시스템이다. 이러한 시스템들은 참가자들이 운영 결정을 내리기 위해 다른 사람들이 기록한 데이터에 의존할 수 있는 충분한 신뢰를 구축한다. 일반적으로 다른 노드들이 암호화된 서명된 거래의 순서를 확인하기 위해 합의 프로토콜을 사용하는 분산 데이터베이스를 사용한다. 시간이 지남에 따라 디지털 서명된 거래를 연결하는 것은 종종 원장의 역사를 사실상 변경 불가능하게 만든다.
공개 키 설명(public key description)
공개 키 또는 검증 키를 사용하기 위해 필요한 모든 메타데이터를 포함하는 DID 문서 내의 데이터 객체이다.
리소스(resource)
[[RFC3986]]에 의해 정의됨: "...'리소스'라는 용어는 URI에 의해 식별될 수 있는 무엇이든 일반적으로 사용된다." 마찬가지로, 어떤 리소스든 DID에 의해 식별되는 DID 주체로서 역할을 할 수 있다.
표현(representation)
HTTP에 대해 [[RFC7231]]에 정의됨: "프로토콜을 통해 쉽게 전달될 수 있는 형식으로 주어진 리소스의 과거, 현재 또는 원하는 상태를 반영하기 위해 의도된 정보, 그리고 표현 메타데이터 집합과 잠재적으로 무한한 표현 데이터 스트림으로 구성된다." DID 문서는 DID 주체를 설명하는 정보의 표현이다. 참조.
표현별 항목(representation-specific entries)
특정 표현에 특별한 의미를 가지는 DID 문서 내의 항목들이다. 및 에서 정의됩니다. 예를 들어, JSON-LD 표현에서의 @context는 표현별 항목이다.
서비스
하나 이상의 서비스 엔드포인트를 통해 DID 주체 또는 관련 엔티티와 의사소통하거나 상호작용하는 수단이다. 예시로는 발견 서비스, 에이전트 서비스, 소셜 네트워킹 서비스, 파일 저장 서비스, 검증 가능한 크리덴셜 저장소 서비스 등이 있다.
서비스 엔드포인트(service endpoint)
서비스가 DID 주체를 대신하여 운영되는 네트워크 주소, 예를 들어 HTTP URL 등이다.
통일 자원 식별자(Uniform Resource Identifier) (URI)
[[RFC3986]]에 의해 정의된 세계적인 웹상의 모든 리소스에 대한 표준 식별자 형식입니다. DID는 URI 체계의 한 종류이다.
검증 가능한 크리덴셜(verifiable credential)
W3C 검증 가능한 크리덴셜 사양 [[VC-DATA-MODEL]]에 의해 정의된 암호학적으로 검증 가능한 디지털 크리덴셜을 위한 표준 데이터 모델 및 표현 형식입니다.
검증 가능한 데이터 레지스트리(verifiable data registry)
탈중앙 식별자 및 DID 문서의 생성, 검증, 갱신, 및/또는 비활성화를 용이하게 하는 시스템이다. 검증 가능한 데이터 레지스트리는 검증 가능한 크리덴셜과 같은 다른 암호학적으로 검증 가능한 데이터 구조에도 사용될 수 있다. 자세한 정보는 W3C 검증 가능한 크리덴셜 사양 [[VC-DATA-MODEL]]을 참조.
검증 가능한 타임스탬프(verifiable timestamp)
제3자가 특정 시점에 데이터 객체가 존재했으며 그 시점 이후로 변경되거나 손상되지 않았음을 검증할 수 있게 하는 검증 가능한 타임스탬프이다. 그 시점 이후에 데이터 무결성이 합리적으로 변경되거나 손상될 수 있다면, 타임스탬프는 검증 가능하지 않다.
검증 방법(verification method)

증명을 독립적으로 검증하기 위해 과정과 함께 사용될 수 있는 매개변수 집합이다. 예를 들어, 암호화된 공개 키는 디지털 서명과 관련하여 검증 방법으로 사용될 수 있으며, 이 경우에는 서명자가 관련 암호화된 개인 키를 소유하고 있었음을 검증한다.

이 정의에서 "검증"과 "증명"은 넓게 적용될 의도이다. 예를 들어, 암호화된 공개 키는 Diffie-Hellman 키 교환 중에 암호화를 위한 공유 대칭 키를 협상하는 데 사용될 수 있다. 이것은 키 합의 과정의 무결성을 보장한다. 따라서 이는 "검증"이나 "증명"이라는 단어를 사용하지 않는 과정 설명에서도 또 다른 유형의 검증 방법이다.

검증 관계(verification relationship)

DID 주체와 검증 방법 사이의 관계 표현이다. 검증 관계의 예는 이다.

범용 고유 식별자(Universally Unique Identifier) (UUID)
[[RFC4122]]에 의해 정의된 전 세계적으로 고유한 식별자의 한 형태이다. UUID는 중앙 집중식 등록 기관을 요구하지 않는다는 점에서 DID와 유사하다. 그러나 UUID는 DID와 달리 해석 가능하거나 암호학적으로 검증 가능하지 않다.
